# Share Card Styles — Design Spec

**Date:** 2026-07-05
**Status:** Approved for planning

## Problem

The Defense Simulator's WhatsApp share card (`api/share-card.js` + `src/components/share/DefenseShareCard.jsx`) currently renders one fixed visual style — Dark Premium (dark navy, blue accents). The certificate download already lets users pick between 3 visual styles (Modern Bold / Academic Prestige / Dark Premium) via `CertificateDownloadModal.jsx`. This spec adds the same kind of choice to the share card: 3 selectable styles, chosen through a picker modal before sharing.

## Goals

- Offer 3 share card styles: **Dark Premium** (existing, unchanged), **Scoreboard** (new), **Academic Prestige** (new).
- Let the user pick a style via a modal before sharing, mirroring `CertificateDownloadModal.jsx`'s pattern (thumbnails + persisted choice).
- Keep the card's fixed 1080×1350 (4:5 portrait) format — no orientation/aspect-ratio choice, unlike the certificate's portrait/landscape toggle. This format was deliberately chosen so the image renders full-width with no letterboxing in WhatsApp chat, and that holds regardless of style.
- Preserve the existing security model: score, topic, and student name are always read server-side from Supabase; the client only ever supplies a cosmetic `style` value, never data that affects the result shown.

## Non-goals

- No new aspect ratios / formats (square, story, landscape).
- No changes to rate limiting, caching behavior (share cards are already generated fresh per request, never cached), or the Vercel serverless function count — everything happens inside the existing `api/share-card.js` function.
- No changes to the certificate flow itself; it's the reference pattern, not something being modified.

## Architecture

Both the certificate and the share card already render server-side and read the score/topic/name from Supabase rather than trusting the client. The share card will follow the certificate's existing multi-style pattern exactly:

- `api/share-card.js` currently has one builder function, `buildCardElement(score, scoreLabel, topic, studentName, logoBase64)`. Rename it to `buildDarkCard` (visuals unchanged) and add two sibling builders:
  - `buildScoreboardCard(score, scoreLabel, topic, studentName, logoBase64)`
  - `buildPrestigeCard(score, scoreLabel, topic, studentName)` (no logo image needed — see below)
- `handler()` reads an optional `style` field from the request body (`'dark' | 'scoreboard' | 'prestige'`, default `'dark'`) and dispatches to the matching builder — the same shape as `api/certificate.js`'s `drawPrestige` / `drawDark` dispatch on its `style` param.
- All three builders stay in `api/share-card.js`, following the precedent already set by `api/certificate.js` (which keeps all of its style-drawing functions in one file rather than splitting across modules).

## Style specifications

### Dark Premium (existing — no visual changes)
Unchanged. Dark navy gradient background, blue score-color accents, existing `fypro-logo.png` fetched from `https://www.fypro.com.ng/fypro-logo.png`.

### Scoreboard (new)
- Background: solid fill using the score's own tier color — green `#16A34A` (≥8), amber `#F59E0B` (5–7), red `#DC2626` (<5) — the same thresholds as `DefenseShareCard.jsx`'s existing `scoreColor()` helper.
- Subtle dot-texture overlay (`radial-gradient` dots) for brand consistency with the rest of the product's workspace texture.
- Huge JetBrains Mono score digit as the dominant element, minimal surrounding chrome.
- Badge (e.g. "DEFENCE READY"): white pill, tier color text.
- Logo: a **new white/mono FYPro logo asset** (see "New asset" below) — solid white shield + "Pro", "FY" kept as a faint ~10% opacity watermark, matching the treatment already used in `fypro-logo-gold.png`. One asset works across all three tier colors, avoiding the need for three separately-colored logo variants.

### Academic Prestige (new)
- Background: ivory `#FFFDF5`, gold `#C9A84C` border and corner ornaments — matches the certificate's Academic Prestige style (`api/certificate.js`'s `drawPrestige`, and `CertificateDownloadModal.jsx`'s preview thumbnail).
- Typography: serif throughout (matching the certificate's prestige treatment), gold accent rule under the score.
- Logo mark: where the certificate uses a circular "FY" monogram, this card uses the **shield icon** from `public/shield-star.svg` instead, inlined directly as SVG in the Satori JSX tree (Satori/`@vercel/og` supports inline SVG paths natively) and recolored gold (`#C9A84C`). No new image asset needed for this — it's an inline path, not a fetched file.

## New asset: `fypro-logo-white.png`

Needed for the Scoreboard style. Generated by recoloring the existing logo's opaque shield+"Pro" regions to solid white while preserving the existing alpha-channel shape and the faint "FY" watermark opacity — the same derivation already used to produce `fypro-logo-gold.png` from the base logo. Not hand-designed from scratch; a scripted recolor (e.g. a short Pillow/Node image-processing script operating on the alpha mask) is sufficient since the shape is identical across every existing color variant.

Two copies are needed, matching the existing duplication pattern for the logo:
- `public/fypro-logo-white.png` — fetched over the network by `api/share-card.js` at request time (same as the existing `fetch('https://www.fypro.com.ng/fypro-logo.png')` call).
- `src/assets/fypro-logo-white.png` — imported directly for the client-side preview bundle (same as `DefenseShareCard.jsx`'s existing `import fyproLogo from '../../assets/fypro-logo.png'`).

## Components

### `ShareCardStyleModal.jsx` (new — `src/components/share/`)
Mirrors `CertificateDownloadModal.jsx` closely:
- 3 style thumbnails (Dark Premium / Scoreboard / Academic Prestige) in a 3-column grid, same active/selected treatment (border highlight + checkmark badge) as the certificate modal.
- No orientation row — format is fixed.
- Selected style persists to `localStorage['share_card_style']`, read on mount with `'dark'` as the fallback default — same pattern as `cert_style`.
- Owns its own loading/error state (moved out of `DefensePrep.jsx`, matching how `CertificateDownloadModal.jsx` is self-contained). Its primary button ("Share to WhatsApp", green, same visual treatment as the current inline button) calls `fetchShareCardBlob(projectId, style)` then `shareToWhatsApp(...)`.
- On successful share, calls back to the parent with the chosen style so the outer preview can update, then closes itself.

### `DefenseShareCard.jsx` (modified — `src/components/share/`)
- Gains a `style` prop (`'dark' | 'scoreboard' | 'prestige'`, default `'dark'`).
- Internally splits into three small render paths, one per style, each mapping 1:1 to its corresponding server-side builder (`buildDarkCard` / `buildScoreboardCard` / `buildPrestigeCard`) so the in-app preview stays visually identical to the server-rendered PNG, per the component's existing doc comment.
- Remains the always-visible preview in the defense summary screen (not hidden inside the modal) — it just now reflects whichever style was last picked (persisted default `'dark'`), consistent with there being no separate "preview" concept in the certificate flow beyond its modal thumbnails.

### `src/lib/shareCard.ts` (modified)
- `fetchShareCardBlob(projectId: string, style: 'dark' | 'scoreboard' | 'prestige' = 'dark')` — adds the `style` param to the existing function signature, threaded into the POST body.

### `src/features/defensePrep/DefensePrep.jsx` (modified)
- Adds `shareStyle` state, initialized from `localStorage.getItem('share_card_style') || 'dark'`.
- The inline `DefenseShareCard` preview passes `style={shareStyle}`.
- The "Share to WhatsApp" button now opens `ShareCardStyleModal` (passing `projectId`, `score`, `scoreLabel`, `topic`, current `shareStyle`) instead of calling `handleShare` directly.
- `shareLoading` / `shareError` state and the `handleShare` function move into `ShareCardStyleModal`; `DefensePrep.jsx` only needs to receive the "style changed" callback to keep its own preview in sync.

## Data flow & validation

- `POST /api/share-card` request body gains an optional `style` field, validated server-side with a plain whitelist array (`const VALID_STYLES = ['dark', 'scoreboard', 'prestige']`) that falls back to `'dark'` for anything not on the list. `api/share-card.js` doesn't use Zod today — unlike `api/ai.js`/`api/auth.js`/`api/payments.js` — so this follows `api/certificate.js`'s existing precedent for validating its own `style` parameter (a plain `validStyles.includes(style)` check) rather than introducing Zod into a file that doesn't otherwise use it.
- Score, topic, and student name continue to come exclusively from `supabaseAdmin` reads inside `handler()` — unchanged from today. `style` is the only new client-supplied value, and it only affects cosmetic rendering, not any data shown on the card, so it introduces no fabrication risk.
- No changes to `rateLimitCheck`, `setCorsHeaders`, or `sendTelegramAlert` usage in `api/share-card.js`.

## Risks

- Satori (the renderer behind `@vercel/og`) supports only a subset of CSS/SVG. The inline gold shield SVG and the Scoreboard style's dot-texture/gradient background need to be verified against actual `ImageResponse` output during implementation, not just against the React preview component, since Satori's rendering behavior can diverge from normal browser CSS.

## Testing / verification

- Generate share cards for all three styles at each score tier (e.g. mock scores 9, 6, 3) to confirm Scoreboard's tier-color background logic and Prestige's fixed gold palette both render correctly via the real `/api/share-card` endpoint.
- Visually confirm the in-app `DefenseShareCard` preview matches the actual downloaded/shared PNG for all three styles, per the component's existing "preview must match server render" invariant.
- Confirm the modal's persisted style choice (`localStorage['share_card_style']`) survives a page reload and correctly pre-selects on next open, matching `cert_style`'s existing behavior.
